## 1. 评价体系

### 1.1 相关性分档标准

- **二手笔记结论（多来源一致）**：工业界做法是人工标注 query 与文档（笔记/网页）的相关度，分为 **4 档：高、中高、中低、低**（有笔记转述为"优/良/中/差"式的表达，但反复出现、可信度更高的措辞是"高、中高、中低、低"）。标注规模：至少标注几十万条 query-doc 二元组，用于训练相关性模型。
- 直觉解释：4 档比二元"相关/不相关"更精细，能区分"完全满足需求"与"部分满足/主题相关但不满足需求"等中间状态，便于后续把标注转成 pointwise 分数（如线性映射到 0~1）或用于 pairwise/listwise 训练。
- 链路位置：这是"相关性"章的基础定义，标注数据同时喂给（a）文本匹配特征模型/BERT 模型的训练（第4节），也用于（b）评价指标 DCG 的 relevance grade 输入（1.2节）。
- 工程要点（二手）：标注体量要求"几十万二元组"级别；训练时会把 4 档标签转换为标量分数（如 0、1/3、2/3、1）供 pointwise 回归或转成 pairwise 偏序对。
- **未找到**：4 档各档的具体书面定义（例如"高=完全满足信息需求且质量高"这类逐档文字定义），仅确认有分档且分 4 级，未找到每档的完整判定标准原文。

### 1.2 相关性评价指标（三层体系）

二手笔记明确提到课程给出的是**三层评价指标体系**，分别对应 pointwise / pairwise / listwise：

- **AUC（pointwise，离线）**：定义为"随机取一个正样本和一个负样本，模型对正样本预测分数大于负样本预测分数的概率"。用于评价相关性模型（如 BERT）在测试集上的整体区分能力。
- **PNR 正逆序比（pairwise，离线）**：Positive-to-Negative Ratio，正序对数 / 逆序对数（模型预测顺序与标注顺序一致为"正序"，相反为"逆序"）。正序越多、逆序越少，说明模型对相对顺序的刻画越准，PNR 越大越好。
- **DCG（listwise，可离线/在线）**：

$$
DCG_p = rel_1 + \sum_{i=2}^{p} \frac{rel_i}{\log_2(i)}
$$

（笔记给出的课程版写法：第 1 位不打折，从第 2 位起除以 $\log_2 i$ 作为位置折损；也有资料写成标准形式 $DCG_p=\sum_{i=1}^p \frac{rel_i}{\log_2(i+1)}$，两种写法在文献中都存在，笔记采用的是前一种"从第2位开始折损"的写法，附录中应以此为准并注明是课程版本。）

$$
nDCG_p = \frac{DCG_p}{IDCG_p}
$$

其中 $IDCG_p$ 是把同一批结果按真实相关性从高到低理想排序后算出的 DCG，用于归一化，使不同 query、不同结果数量的 DCG 可比。

- 直觉解释：AUC 只看"谁比谁相关"的整体统计概率，不关心具体名次；PNR 直接统计有多少对顺序判断对/错；DCG/nDCG 则真正引入"位置"概念——排名越靠前的相关文档，对整体分数贡献的折扣越小，符合用户从上往下看结果的行为假设。
- 链路位置：AUC/PNR 主要用在**模型迭代阶段的离线评估**（例如相关性 BERT 模型上线前的效果验证）；DCG/nDCG 既可用离线人工标注计算，也可结合线上小流量的排序结果计算，是最终衡量"整条链路排序质量"的核心指标。
- 工程要点：三层指标一般配合使用——AUC/PNR 用于快速离线迭代排查模型问题，DCG/nDCG 更贴近线上真实体验但计算和标注成本更高。

### 1.3 用户满意度的行为信号

课程"什么决定用户满意度？"一节，二手笔记确认提到的信号：

- **有点比（查询词点击率）**：`有点比 = 查询词点击总次数 / 搜索总次数`，取值 ≤ 1，笔记给出经验值"约 70% 左右"。含义：搜索之后用户是否至少点了一个结果；局限是"点击"不代表真正满意（可能点开又秒退）。
- **首屏有点比**：`首屏有点比 = 首屏点击总次数 / 搜索总次数`，比"有点比"更严格（只统计首屏内的点击），数值上 **首屏有点比 < 有点比**，笔记给出经验值"约 60% 左右"。含义：反映排序头部结果是否足够好，避免用户翻页/下滑才能找到满意结果。
- **（主动）换词率**：用户搜索后没找到想要的内容，会更换查询词重新搜索；换词率定义为"一定时间窗口内，前后两个查询词编辑距离很小（即认定为同一意图的换词）"的比例。换词率高说明当前查询词的排序结果没能满足需求，是负向信号。
- **退出率**：二手笔记只是笼统提到"改进排序策略通常会同时提升有点比、首屏有点比、降低换词率"这类中间指标组合，但**没有找到退出率的独立定义原文**，视为"未找到，仅有章节标题式的旁证"。
- 局限性讨论（二手笔记概括）：这些行为信号都是"满意度的代理指标"，各自都有偏差——点击率会被标题党/位置偏差(position bias)影响；首屏点击率对结果条数、卡片大小敏感；换词率对多轮探索性搜索（本来就要多次尝试）不友好。课程强调这些指标需要组合使用，不能只看单一指标。
- 链路位置：这些信号是"验证相关性/满意度提升是否奏效"的**在线 AB 实验指标**，与第1.2节的离线 DCG/AUC 构成"离线-在线"两层评价体系。

## 2. 文本匹配分数

课程小节"文本匹配分数"（二手笔记标题："相关性03：文本匹配（TF-IDF、BM25、词距）"）。

### 2.1 TF-IDF

- 直觉：词频（TF，词在文档中出现的次数）越高，该词对文档主题的代表性越强；但如果一个词几乎在所有文档里都出现（如"的"、"是"），区分度就低，用逆文档频率（IDF）打压这类词的权重。
- 课程角色：TF-IDF 是"传统文本匹配打分"体系中最基础的一种，笔记称之为"词匹配分数"的代表，和 BM25 并列，作为早期（2020年前）排序阶段人工设计特征之一，现在（笔记原话）"排序阶段普遍抛弃了文本匹配方法改用 BERT，文本匹配只保留在文本召回环节做候选集筛选"。

### 2.2 BM25

- 二手笔记确认的公式要点（BM25 的标准形式，课程沿用信息检索经典定义）：

$$
\text{score}(q, d) = \sum_{t \in q} IDF(t) \cdot \frac{f(t,d)\cdot (k_1+1)}{f(t,d) + k_1\left(1-b+b\cdot \frac{|d|}{avgdl}\right)}
$$

- **k1（词频饱和度参数）**：经验取值区间 1.2~2.0。作用是让"词频对分数的贡献"呈现非线性饱和——k1 越小，饱和越快（词出现几次之后再增加也不会明显加分）；k1 越大，词频的边际贡献衰减越慢。课程强调 BM25 相对 TF-IDF 的核心改进之一，就是发现"词频与相关性的关系是非线性的，单个词的贡献不会随词频无限增长"。
- **b（文档长度归一化参数）**：取值 0~1。b=1 表示完全按文档长度缩放词频（长文档中的高词频会被压得更低）；b=0 表示不考虑文档长度。作用是校正长文档天然因为词多而在词频上占便宜的问题。
- IDF 部分与常规 IDF 计算方式一致，作为该词在查询里的固有权重，再对查询中所有词项加权求和得到 query-doc 匹配分数。

### 2.3 词权重在匹配分数中的用法

- 匹配分数公式里 IDF 项本身就承担了"词的全局权重"角色；此外查询词处理阶段算出的 Term Weight（见第3.2节）也可以作为加权系数，对核心词（实体词、专名）的匹配分数做加权放大，对停用词/虚词类的匹配贡献做抑制，笔记里提到"可以把 Term Weight 拿来做 term 的激励"。

### 2.4 其他文本匹配特征：词距/紧密度（Proximity）

- 二手笔记确认课程提到 **OkaTP、BM25TP** 两个词距类匹配分数（"词距分数"）。
- 直觉解释：如果 query 中两个词在语义上紧密相关（比如"深度学习"里的"深度"和"学习"），那么在候选文档中，这两个词如果挨得很近（甚至连续出现），说明文档更可能是在讨论这个整体概念，而不是两个词分别出现在不相关的上下文里——因此词距越近，相关性打分应越高。
- 链路位置与地位：文本匹配分数（TF-IDF/BM25/词距）曾经是排序阶段的核心特征（线性模型或树模型的输入特征之一），二手笔记明确指出"2020年后搜索排序普遍放弃文本匹配方法转向 BERT，文本匹配分数现在主要用于**文本召回**阶段的候选集筛选，而不是精排打分"。

## 3. 查询词处理经典链路

课程第3章共6小节：分词(字典匹配+新词发现) / 分词(深度学习) / 词权重 / 类目识别 / 意图识别 / 查询词改写。二手笔记对该章节覆盖程度弱于第1、2章（没找到专门逐节解读的博客，只有零散提及和"概要04：搜索引擎的链路"里的总览式描述），故本节多数条目标注了置信度。

### 3.1 分词

- **基于字典匹配的方法**：课程标题明确包含"字典匹配"和"新词发现"，属于经典规则分词范畴。二手笔记（非王树森专属，但作为分词通识背景）确认的相关方法：正向最大匹配、逆向最大匹配、双向最大匹配——即用一个机器词典，从字符串开头（或结尾）尽量匹配最长的词。
  - **未找到**：王树森课程里"字典匹配"具体使用哪种最大匹配变体、词典规模、冲突消解规则等细节，仅能确认小节标题存在。
- **新词发现**：二手笔记里提到与新词发现相关的通用方法是"预分词类似最大匹配，将 n-gram 片段拼接成尽量长的候选词"，再结合互信息/左右熵等统计量筛选候选新词——但这是新词发现领域的通识描述，**未能确认这是王树森课程原文的具体讲法**，只能作为背景参考。
- **基于深度学习的方法**：二手笔记提到分词可以建模为**序列标注问题**，可使用 BERT 等大规模预训练模型，并可通过知识蒸馏把大模型的分词能力迁移到小模型上以降低线上推理成本。这与课程"词权重"一节中提到的序列标注技术路线（BiLSTM/BiLSTM+CRF/BERT）是同一技术家族，但**未找到专门针对"深度学习分词"小节的原文细节**（如具体标注体系 BMES、具体模型结构对比等）。

### 3.2 词权重（Term Weight）

- 二手笔记（多来源交叉）给出的定义和建模方式：
  - Term Weight 用于刻画 query 中每个词对整体查询意图的重要程度，用于后续召回/排序中区分"核心词"与"修饰词/虚词"。
  - **建模方式**：可建模为**序列标注任务**或**回归任务**。序列标注常用 BiLSTM、BiLSTM+CRF、BERT 等模型，输出层可以接分类头（离散档位）或回归头（连续分数）。回归任务的一种做法是：用深度模型（BERT 或 BiLSTM+Attention）生成上下文相关的词向量，再接线性回归层预测权重分数。
  - **权重档位（二手笔记给出的示例三档，注意这是通识笔记的补充说明，非确认的王树森原文数值）**：高权重（>0.66，核心实体/专有名词）、中权重（0.33~0.66，限定词/动词/版本号类）、低权重（<0.33，停用词/语气词）。此档位切分**未在王树森专属笔记中直接确认**，建议附录中标注为"业界常见做法，具体阈值课程原文未确认"。
- 链路位置：Term Weight 结果一方面用于第2.3节文本匹配分数的加权，另一方面供召回阶段做"控召回范围"（核心词必须命中，虚词可忽略）。

### 3.3 类目识别

- 二手笔记确认：类目识别是查询处理阶段的重要步骤，用于识别 query 所属的类目/品类信息，以提升检索准确性（多出现在电商类场景的表述中，例如"商品归入某大类下的小类目"）。
- **未找到**：王树森课程具体采用的类目体系（是否为电商多级类目还是小红书笔记类目/话题体系）、类目识别的模型结构（多分类 softmax？层级分类？）等细节，仅确认有该小节标题及泛化的"类目识别提升检索准确性"这一句话，未获得课程专属的模型细节。

### 3.4 意图识别

- 二手笔记确认：意图识别用于"识别 query 中的时效性意图和地域性意图"，例如判断用户是否在找"最新"的内容（时效性）或"本地/附近"的内容（地域性），从而让下游召回/排序做相应调整（如提升近期发布内容权重，或结合地理位置召回）。
- **未找到**：完整的意图分类体系表（课程是否还包含导航型/交易型/信息型等经典意图分类）、具体使用的模型（是文本分类模型还是规则+分类混合），只能确认"时效性、地域性"这两类意图被明确提及。

### 3.5 查询词改写

- 二手笔记确认：查询词改写模块包含**纠错（拼写纠错）**、**同义词/近义词改写**、**查询扩展**这几类经典能力，目的是"改进用户输入、增强搜索匹配度"。查询扩展的通识定义是"通过添加同义词或相关词块/短语扩展原始查询，帮助召回更多相关但字面不完全匹配的文档"。
- **未找到**：王树森课程中同义词库的构建方法（是词向量聚类还是知识图谱/日志挖掘）、纠错模型的具体架构（编辑距离规则 or seq2seq 纠错模型）等实现细节，仅能确认查询词改写包含"改写、纠错、（同义词）扩展"这几个子能力的框架性描述，且课程标题明确排除与 LLM 相关内容属于经典方法范畴。

## 4. 相关性 BERT 模型

课程有专门两节："相关性BERT模型及其推理"、"相关性BERT模型的训练"，是本次调研中二手笔记覆盖最详细的部分之一。

### 4.1 训练方式（四阶段流程）

二手笔记（多来源一致）明确给出课程的训练流程分 **4 个阶段**，无论是交叉 BERT 还是双塔 BERT，训练方法相同：

1. **预训练（Pretrain）**：用 MLM（Masked Language Model）等自监督任务预训练。笔记提到"直接用开源模型效果也不差，但最好用自己搜索引擎的文档库做预训练，效果更好"（即领域自适应预训练 domain-adaptive pretraining）。
2. **后预训练（Post-pretrain）**：笔记特别强调这是"新技术"——利用用户的点击、停留等交互日志作为弱监督信号继续预训练，原理是"相关性越好的文档越可能被点击/交互"。具体做法：输入为 `[q, d]` 的 pair，同时使用点击标签 `y` 和 MLM 两种监督信号联合训练。
3. **微调（Fine-tuning）**：使用人工标注的相关性数据（即第1.1节的4档标注）训练，笔记提到会把分档标注**转换为 0~1 之间的分数**作为回归/分类目标，这一步是 pointwise 的有监督训练。
4. **蒸馏（Distillation）**：把训练好的大模型（Teacher，通常是层数更多、更准的交叉 BERT）蒸馏到一个更小的模型（Student）中，用于降低线上推理成本。

- 直觉解释：这套四段式训练本质是"先用大规模无标注文本学语言 → 再用大规模弱标注点击数据学'相关性相关的语言模式' → 再用小规模高质量人工标注数据精调到目标任务 → 最后把知识压缩进小模型上线"，是搜索工业界应对"标注数据稀缺但点击数据海量"矛盾的典型解法。
- **未找到**：微调阶段具体使用 pointwise 回归还是 pairwise ranking loss（如 hinge loss/pairwise cross-entropy）的明确说明——笔记只提到"标注转成 0~1 分数"更像 pointwise 回归的描述，但没有确认是否同时使用了 pairwise 方式，故不应在附录中断言这是纯 pairwise 或纯 pointwise，建议表述为"至少包含 pointwise 回归，是否叠加 pairwise 未确认"。

### 4.2 推理优化手段

- **双塔 BERT（Dual-tower / 双塔）**：二手笔记明确描述——双塔 BERT 准确性较低但计算量小，适用于**粗排/召回海选**环节；做法是离线预先算好每篇文档 `d` 的向量表征 `z_d`，存入 KV/哈希表，线上只需实时计算 query 的向量表征 `x_q`，再计算内积 `⟨x_q, z_d⟩` 作为 (q,d) 的相关性分数，从而避免线上对每个文档做一次完整的 BERT 前向计算。
- **交叉 BERT（Cross-encoder）**：query 和文档拼接后一起输入 BERT 做交互式打分，准确性更高但计算量大，笔记确认**精排环节常用 4~12 层的交叉 BERT**，**粗排环节常用 2~4 层的交叉 BERT（或者用双塔 BERT代替）**——即链路里从粗排到精排，逐步用更深、更准但更贵的交叉 BERT，候选集也随之从多到少收窄。
- **模型量化**：笔记提到"用模型量化技术，把 float32 转成 int8，降低推理成本"，是另一条独立于蒸馏的推理加速手段。
- **知识蒸馏**：如4.1节所述，是主要的模型压缩技术之一，把 Teacher 模型的知识压缩进更小的 Student 模型。
- **降低序列长度/截断**：笔记提到"设置较小的 token 数量上限，将长文档替换成摘要"，是另一种降低推理成本的工程手段（减少 BERT 输入长度从而降低计算量）。
- **缓存**：文本匹配、BERT 推理结果的缓存策略在"缓存召回"一节中有专门讨论（见第5.3节），但**未找到**"相关性 BERT 打分结果缓存"这一专属机制的课程原文细节，只能确认双塔向量的离线预计算+哈希表存储属于广义的"缓存"思路。
- 链路位置总结（二手笔记综合）：粗排阶段用双塔 BERT（或浅层交叉 BERT，2~4层）做初步海选，精排阶段用较深的交叉 BERT（4~12层）做精细打分；模型量化、蒸馏、截断输入长度是三类独立的推理成本优化手段，可叠加使用。

## 5. 召回补充

课程第4章：倒排索引和文本召回 / 向量召回 / 缓存召回。

### 5.1 缓存召回

- 二手笔记明确定义：查询词处理（QP）流程中设有缓存——**如果 query 命中缓存，直接读取缓存中已经算好的 QP 结果（分词、词权重、类目、意图等），不需要重新调用各个模型服务实时计算**。
- 为什么用：笔记给出的理由是"对于通用搜索引擎，几百万个高频 query 就能覆盖每天的大部分搜索请求"——即查询词分布高度长尾/头部集中，对高频 query 做缓存可以用很小的存储成本换取巨大的在线计算量节省和延迟降低。
- **失效与更新机制**：**未找到**课程原文对缓存失效策略（TTL、版本号、增量更新触发条件等）的具体描述，只笼统确认"命中缓存则跳过计算"这一基本逻辑，缓存更新细节标注为"未找到，仅有概念性描述"。

### 5.2 文本召回与倒排索引

- 二手笔记描述：文本召回是搜索引擎的**主通道**，基于**倒排索引 + 召回规则**实现——倒排索引的 key 通常是"冬季"、"卫衣"、"推荐"这类常用词/实体词，词表规模不算特别大（相对全网页面数量而言）；查询时先做分词/改写得到若干 term，再通过倒排索引取出包含这些 term 的文档集合，结合召回规则（比如必须命中核心词）过滤候选集。
- 与向量召回的互补关系：文本召回属于**主通道**，向量召回（双塔模型）属于**旁路通道**。二手笔记的理解是——文本召回保证"字面/关键词匹配"的精确度和可解释性（是否命中核心词，规则可控），向量召回则能召回字面不匹配但语义相关的文档（同义表达、跨词汇的语义相近内容），两者合并去重后一起送入排序，从而同时兼顾"准确匹配"和"语义泛化"的召回覆盖率。
- **未找到**：向量召回具体的双塔模型结构细节（是否与相关性 BERT 的双塔共享参数）、ANN 检索引擎选型等工程细节，仅确认"向量召回=双塔模型"这一对应关系。

## 6. 排序融合

课程第5章：排序的原理 / 融合模型的训练方法。

### 6.1 融合模型

- 二手笔记明确描述融合逻辑：**排序需要综合计算相关性、点击率(CTR)、内容质量、时效性等多个分数，然后用"规则"或"模型"两种方式之一把这些分数融合，给出最终排序分。**
- 融合公式形态：笔记给出的阶段性建议是——**"初期最好用人工设定的融合公式（如加权和形式），不断根据线上表现迭代权重；到后期数据和经验积累充分后，再改用模型融合（比如用一个模型直接学习如何组合这些分数，而不是手工调权重）"**。也就是说课程给出的是"加权和 → 模型融合"的演进路径，而不是从一开始就用复杂模型。
  - 加权和的直觉形态可表示为（课程未给出精确系数公式，此处为通识加权和写法，仅作示意）：
    $$
    \text{FinalScore}(q,d) = w_{rel}\cdot \text{Relevance}(q,d) + w_{ctr}\cdot \text{CTR}(q,d) + w_{qual}\cdot \text{Quality}(d) + w_{time}\cdot \text{Freshness}(d)
    $$
  - **未找到**：课程给出的具体加权系数、或"模型融合"阶段具体采用什么模型结构（LR/GBDT/深度多目标网络等），只能确认存在"规则→模型"两阶段融合的思路表述。
- CTR 的定义（二手笔记确认）："点击率(CTR) 指文档曝光的前提下，用户点击该文档的概率"，是融合分数中的一路输入。
- 直觉解释：单一目标（比如只优化相关性）可能牺牲用户体验的其他维度（比如推荐了相关但过时的内容，或相关但质量低的内容），所以需要多目标融合；融合的核心矛盾在于"人工经验规则"简单可控但难以拟合复杂的非线性关系，"模型融合"效果上限更高但需要更多数据和更复杂的在线serving/迭代成本，因此课程建议按业务发展阶段循序切换。

### 6.2 融合模型的训练方法

## 7. 查询词推荐

课程第6章：查询词推荐的场景 / 查询词推荐的召回 / 查询词推荐的排序。

### 7.1 场景

- 二手笔记区分了两类容易混淆的场景：
  - **SUG（搜索联想/下拉提示）**：用户尚未点击搜索、还在输入框打字时，根据已输入的前缀实时提示候选 query（例如输入"深度学习"提示"深度学习框架""深度学习教程"）。笔记强调 **SUG 是独立于搜索结果页之外的服务**，基于用户已输入的文字去匹配高热度 query，返回补全后的完整 query。
  - **相关搜索**：出现在**搜索结果页底部**，是搜索引擎"query 分析/变换"能力的一部分，为用户在看完当前结果后提供进一步探索的相关 query。
  - **底部推荐**：课程标题中提到的第三个场景，**未找到**二手笔记对其与"相关搜索"的具体区分描述（有可能"相关搜索"与"底部推荐"在课程中是同一概念的不同表述，也可能是两个独立位置），此处不作强行区分，标注为"未找到明确定义，仅知场景名称"。

### 7.2 召回方法

- **SUG 召回**：二手笔记确认使用 **Trie（前缀树）** 结构做前缀匹配——将历史高频 query 建成 Trie 树，用户输入的前缀去 Trie 中做前缀检索，召回 TopN（笔记提到 N 通常取 10 左右）候选 query。
- 热门 query 的挖掘方法（通识背景，非专属确认）：通过统计搜索日志中 query 字符串出现频次，取频次最高的 query 作为热门候选池，是 Trie 树召回候选的来源之一。
- **相关搜索的召回方法**：**未找到**课程专属描述（例如是否基于 query 共现统计、query 向量相似度，还是与向量召回复用同一套双塔模型），仅能确认相关搜索属于"query 分析/变换"体系下的产物，具体召回算法未确认。

### 7.3 排序方法

- 二手笔记确认 SUG 场景下的排序思路：**用曝光日志和用户点击日志拼接成训练样本，训练一个（笔记原文提到）LR 模型预测 SUG 候选词的点击率(CTR)，再按预测 CTR 对候选 query 排序**后返回给用户。也提到过一般流程是"召回 TopN → 拼特征 → 模型打分排序 → 返回结果"这一通用范式。
- **未找到**：相关搜索/底部推荐场景的专属排序方法描述，只能确认其大概率遵循"召回+排序"这一通用范式，但没有找到课程给出的具体排序特征或模型选择的原文佐证。
